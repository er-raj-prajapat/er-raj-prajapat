## The short version

I build tools that sit at the **intersection of developer experience and AI systems** — specifically where trust, contracts, and behavioral guarantees matter most.

Most developers write code that *does things*. I prefer to build systems that *govern how things get done* — enforcement layers, audit trails, semantic detectors, behavioral contracts.

If AI is changing what's **possible**, I'm focused on what's **permissible and verifiable**.

**Policy-as-code for AI coding agents**

`trust infrastructure` · `policy engine` · `audit trail`

**Detect silent behavioral drift in APIs**

Behavioral fingerprinting for third-party APIs. Snapshots schemas, status codes, headers, latency. Produces semantic diffs: not *"is it up?"* but *"did it silently change?"* — the blind spot between changelogs and incidents.

`behavioral contracts` · `semantic diff` · `CI-ready`

## How I think about problems

```
Problem space I operate in:
  ├── AI systems moving faster than trust infrastructure can keep up
  ├── APIs changing silently while teams rely on undocumented behavior
  ├── Developer tooling that adds friction instead of removing it
  └── The gap between "it works" and "it can be reasoned about"

Default instincts:
  ├── Systems first, features second
  ├── The best interface is one you don't notice
  ├── Audit trails are a product feature, not an ops concern
  └── Implicit contracts should be made explicit and machine-readable
```
